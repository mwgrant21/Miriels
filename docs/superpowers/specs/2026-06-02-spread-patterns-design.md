# Spread Patterns — Design Spec
**Date:** 2026-06-02  
**Status:** Approved

---

## Overview

Add 9 new tarot spread patterns to the app, reorganize the spread selector into category tabs, and update Miriel's Reader's Choice AI to pick from all spreads. The implementation is data-driven: all spread metadata (category, label, slots) lives in the `SPREADS` object, and buttons are generated from it rather than hardcoded in HTML.

## Data Structure Change (`public/app.js`)

### SPREADS object — new format

Each entry changes from `key → array` to `key → { category, label, slots, special? }`:

```js
const SPREADS = {
  'single': {
    category: 'general',
    label: 'Single',
    slots: [{ label: 'Card', position: '' }]
  },
  'horseshoe': {
    category: 'general',
    label: 'Horseshoe',
    slots: [
      { label: 'Past',              position: 'past' },
      { label: 'Present',           position: 'present' },
      { label: 'Hidden Influences', position: 'hidden' },
      { label: 'Obstacles',         position: 'obstacles' },
      { label: 'External Forces',   position: 'external' },
      { label: 'Best Action',       position: 'action' },
      { label: 'Outcome',           position: 'outcome' }
    ]
  },
  'reader-choice': {
    category: 'general',
    label: '✦ Reader\'s Choice',
    special: 'reader-choice',
    slots: []  // filled dynamically by AI
  },
  'compatibility': {
    category: 'relationship',
    label: '♥ Compatibility',
    special: 'modal',
    slots: [ /* existing slots */ ]
  },
  // ... all other spreads
};
```

The `special` field preserves existing custom behaviors:
- `'reader-choice'` → triggers AI spread selection flow
- `'modal'` → opens the name-entry modal before drawing

### All SPREADS[key] callsites

Every reference to `SPREADS[key]` as an array becomes `SPREADS[key].slots`. Approximately 12 sites in `app.js`:
- `const spreadDef = SPREADS[currentSpread]` → `SPREADS[currentSpread].slots`
- `SPREADS['compatibility'][0].label` → `SPREADS['compatibility'].slots[0].label`
- `if (data.spread && SPREADS[data.spread])` → stays the same (object is still truthy)
- etc.

## Tab UI (`public/app.js` + `public/index.html`)

### HTML changes

Replace the `#spread-select` div contents with three tab buttons only. The spread buttons are removed from HTML entirely — they are generated by JS.

```html
<div id="spread-tabs" class="tab-bar">
  <button class="tab active" data-tab="general">General</button>
  <button class="tab" data-tab="relationship">Relationship</button>
  <button class="tab" data-tab="spiritual">Spiritual</button>
</div>
<div class="btn-group" id="spread-select">
  <!-- generated by initSpreadButtons() -->
</div>
```

### Button generation

`initSpreadButtons()` runs on page load. It iterates `Object.entries(SPREADS)` and creates a button for each entry with `data-value` and `data-category` attributes. Special spreads get their existing CSS classes (`btn-reader-choice`, `btn-compat`).

### Tab behavior

- **Tab click**: hides all `#spread-select .btn` not matching `data-category`, marks tab active. Preserves current spread highlight.
- **Spread button click**: existing behavior unchanged (sets `currentSpread`, marks button active).
- **Reader's Choice result** (`chosenSpread` resolved): sets active spread button AND calls `activateTab(SPREADS[chosenSpread].category)` so the highlighted button is visible. This makes it feel like Miriel took control — the tab switches to wherever her choice lives.
- **Default on load**: General tab active, Single selected.

### `activateTab(category)` helper

New function — switches the active tab to the given category and filters the visible buttons. Called from the Reader's Choice flow after the spread is resolved.

## CSS Layouts (`public/style.css`)

Five new spread grid types. All others (yes-no, rel-cross, soulmates, rel-future) use the existing default flex row.

### Lookup table (`public/app.js`)

Replace the `renderSpread` if/else chain with a lookup table:

```js
const SPREAD_LAYOUTS = {
  'celtic':        { gridClass: 'celtic-grid',   cardClasses: CELTIC_CLASSES },
  'six-card':      { gridClass: 'six-grid',       cardClasses: SIX_CARD_CLASSES },
  'compatibility': { gridClass: 'six-grid',       cardClasses: SIX_CARD_CLASSES },
  'nine-card':     { gridClass: 'nine-grid',      cardClasses: NINE_CARD_CLASSES },
  'horseshoe':     { gridClass: 'horseshoe-grid', cardClasses: HORSESHOE_CLASSES },
  'year-ahead':    { gridClass: 'year-grid',      cardClasses: YEAR_CLASSES },
  'chakra':        { gridClass: 'chakra-grid',    cardClasses: CHAKRA_CLASSES },
  'decision':      { gridClass: 'decision-grid',  cardClasses: DECISION_CLASSES },
  'star':          { gridClass: 'star-grid',      cardClasses: STAR_CLASSES },
};
```

`renderSpread` looks up layout, applies grid class and per-card CSS classes if found, falls back to default flex row otherwise.

### New CSS grid types

**`horseshoe-grid`** — flex row, cards get height CSS classes to create the arc:
- Classes: `hs-1` through `hs-7`, height gradient: 30px · 38px · 46px · 52px · 46px · 38px · 30px

**`year-grid`** — CSS grid, 4 columns × 3 rows, uniform card size, position label overlaid on each card

**`chakra-grid`** — flex column, centered; each card slot gets a chakra accent class (`chakra-root`, `chakra-sacral`, etc.) for a subtle left-border color (red → orange → yellow → green → blue → indigo → violet)

**`decision-grid`** — 3-column CSS grid: Situation (col 1, spans 2 rows, tall) · Path A row / Path B row (col 2) · Outcome (col 3, spans 2 rows, tall). Path A cards at full opacity, Path B cards at 85% opacity.

**`star-grid`** — fixed-size container (160×160px), cards absolutely positioned at pentagram points. Spirit (slot 5, drawn last) maps to the top point per tradition. Visual position mapping by slot index: Earth (1) → lower-left, Air (2) → upper-right, Fire (3) → lower-right, Water (4) → upper-left, Spirit (5) → top center. STAR_CLASSES encodes this mapping so `renderSpread` can assign positions by index.

## suggest-spread Server Update (`server.js`)

### `spreadMenu` string

Expanded to include all 18 spreads with Miriel-voiced descriptions. Format follows the existing pattern: `key — n cards. Best for: ...`. New entries:

```
yes-no — three cards. Best for: when someone needs a direct answer and doesn't want the long view.
horseshoe — seven cards. Best for: situations with a clear narrative arc where hidden forces and obstacles matter.
year-ahead — twelve cards, one per month. Best for: January readings, birthdays, or major life transitions — when someone needs the broad shape of a season ahead.
decision — six cards. Best for: genuine crossroads where two real options exist and the person needs to feel both paths before choosing.
rel-cross — six cards. Best for: understanding the dynamic between two people from both sides, with where it's heading.
soulmates — six cards. Best for: questions about deep connection, soul-level bonds, or past-life resonance.
rel-future — six cards. Best for: a relationship that needs directional clarity — not just what's happening but where it's going.
chakra — seven cards. Best for: questions about the body, energy blocks, or when something physical or emotional feels stuck without explanation.
star — five cards. Best for: elemental questions, spiritual grounding, or readings where the person wants to understand which forces are in play.
```

### `valid` array

Expanded from 7 to 18 keys: all existing keys plus `yes-no`, `horseshoe`, `year-ahead`, `decision`, `rel-cross`, `soulmates`, `rel-future`, `chakra`, `star`.

## Spread Label Display in spreadLabels / reading save

`app.js` has a `spreadLabels` lookup (line ~1465) used when saving readings. This gets entries for all 9 new spread keys mapping to their display names. The reading save and history replay logic uses `currentSpread` to restore the correct tab when replaying a saved reading — a `activateTab` call is added to the replay path.
